# Deployment Improvements

This document describes the changes made to the Docker infrastructure of the Order Management System as part of Task 4 (Option B) of the assignment.

---

## Summary of Changes

| File | Change |
|------|--------|
| `backend/Dockerfile` | Switched to `node:18-alpine`, added layer caching, non-root user |
| `frontend/Dockerfile` | Multi-stage build with `nginx:alpine`, React Router support |
| `backend/.dockerignore` | New file — excludes `node_modules`, `.env` |
| `frontend/.dockerignore` | New file — excludes `node_modules`, `build`, `.env` |
| `docker-compose.yml` | Added healthcheck, proper `depends_on`, env vars, restart policies |
| `backend/src/config/db.js` | Reads DB credentials from env vars instead of hardcoding |

---

## Detailed Changes

### 1. Backend Dockerfile

**Before:**
```dockerfile
FROM node:18
WORKDIR /app
COPY . .
RUN npm install
EXPOSE 3001
CMD ["npm", "start"]
```

**After:**
```dockerfile
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY src ./src
USER node
EXPOSE 3001
CMD ["node", "src/index.js"]
```

**Why:**
- **`node:18-alpine`** — Alpine-based images are ~150MB vs ~900MB for the full Debian image. Smaller attack surface and faster pulls.
- **Copy `package*.json` first** — Docker caches layers. If only source code changes (not `package.json`), npm install is skipped on rebuild, making builds significantly faster.
- **`npm ci` instead of `npm install`** — `npm ci` installs exactly what's in `package-lock.json` for reproducible builds. `npm install` can silently upgrade minor/patch versions.
- **`--only=production`** — Dev dependencies (like `nodemon`) are not installed in the production image.
- **`USER node`** — Runs the process as the non-root `node` system user. If the container is compromised, the attacker has limited privileges.
- **`node src/index.js` directly** — Avoids the overhead of `npm` as a process wrapper; signals (SIGTERM) go directly to the Node process for graceful shutdown.

### 2. Frontend Dockerfile

**Before:**
```dockerfile
FROM node:18
WORKDIR /app
COPY . .
RUN npm install
EXPOSE 3000
CMD ["npm", "start"]
```

**After (multi-stage):**
```dockerfile
# Stage 1: Build
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY public ./public
COPY src ./src
RUN npm run build

# Stage 2: Serve
FROM nginx:alpine AS production
COPY --from=builder /app/build /usr/share/nginx/html
# nginx config for React Router support
EXPOSE 3000
CMD ["nginx", "-g", "daemon off;"]
```

**Why:**
- **Multi-stage build** — The Node.js toolchain (used only to build) is not included in the final image. Only the compiled static HTML/CSS/JS files are shipped. The final image is ~25MB instead of ~1GB.
- **`nginx:alpine`** — nginx is a production-grade static file server. `react-scripts start` (CRA dev server) is for development only — it has no optimisations, no proper caching headers, no gzip, and is not intended for production traffic.
- **React Router fallback** — The nginx config includes `try_files $uri $uri/ /index.html`, which correctly handles client-side route refreshes (without this, refreshing on `/orders` returns a 404).

### 3. .dockerignore Files

New `.dockerignore` files were added for both `backend/` and `frontend/`.

**Why:**
- Without `.dockerignore`, Docker sends everything in the build context to the daemon — including `node_modules` (potentially hundreds of MBs) and `.env` files. This makes builds slow and risks leaking secrets.
- The `node_modules` folder is always regenerated by `npm ci` inside the container, so it should never be copied from the host.

### 4. docker-compose.yml

**Key changes:**

#### Healthcheck on `db`
```yaml
healthcheck:
  test: ["CMD-SHELL", "pg_isready -U ${DB_USER:-admin} -d ${DB_NAME:-orderdb}"]
  interval: 5s
  timeout: 5s
  retries: 10
```
Previously, the `db` service had no healthcheck. PostgreSQL takes a few seconds to initialise on a cold start. Without a healthcheck, the backend starts immediately after the `db` container starts (not after Postgres is *ready to accept connections*), causing connection errors and crash loops.

#### `depends_on` with `condition: service_healthy`
```yaml
backend:
  depends_on:
    db:
      condition: service_healthy
```
The old `depends_on: - db` only waited for the container to *start*, not for it to be *ready*. The new form waits for the healthcheck to pass before starting the backend. This eliminates the race condition on `docker compose up --build`.

#### Environment Variables
```yaml
environment:
  DB_USER: ${DB_USER:-admin}
  DB_PASSWORD: ${DB_PASSWORD:-admin123}
  DB_NAME: ${DB_NAME:-orderdb}
```
Credentials are now passed via environment variables using shell variable substitution with fallback defaults (`:-admin`). For production, set `DB_USER`, `DB_PASSWORD`, `DB_NAME` as actual environment variables or via a `.env` file (not committed to git). The backend `db.js` was updated to read `process.env.DB_USER`, etc., rather than hardcoding values.

#### `restart: unless-stopped`
Added to all services so containers automatically restart after crashes or host reboots — standard practice for production deployments.

#### `postgres:15-alpine`
Switched from `postgres:15` to `postgres:15-alpine` for a smaller database image.

## How to Run

```bash
# Start all services (rebuilds images automatically)
docker compose up --build

# To override credentials for production:
DB_USER=myuser DB_PASSWORD=securepass DB_NAME=orders docker compose up
```

Access the services:
- **Frontend:** http://localhost:3000
- **Backend API:** http://localhost:3001/api
- **Health check:** http://localhost:3001/api/health
